## 四、回归根因分析

### 回归 1：Case 8 散步鞋 → 0 cards

第一轮返回 3 个鞋类结果，第二轮 0 cards。可能原因：
- session 上下文的 `target_sub_categories` 从其他 session 泄漏（`session_id="trace_case2"` 可能污染了全局 session 状态）
- 或 `infer_product_type("运动鞋")` 在新的 session 上下文中被错误覆盖

### 回归 2：Case 19 游戏手机 Turn 15-19 → 0 cards

"就小米17 Ultra"后连续 0 cards。可能原因：
- `target_sub_categories` 从 session 兜底为错误值
- 或 LLM 路由器的 query 覆盖了品类信息

### 回归 3：Case 1 面霜 Turn 3 → 0 cards

"价位在300元左右"后 0 cards。可能原因：
- session 上下文的 `target_sub_categories` 被设为 `['精华']`（来自 Case 11 的测试）
- 面霜不在"精华"子品类中 → 过滤为空

## 五、核心结论

### 改善显著

- **PC 装机多轮稳定性**：Case 4 和 Case 12 从 PARTIAL → PASS，7 轮对话全部正确路由到 `generate_pc_build_plan`
- **品类锁定**：Case 2/9/15/17 的 session 品类上下文传递生效，不再品类漂移
- **should_skip_llm_route 修复**：PC 装机 followup 不再被跳过 LLM

### 回归需关注

- **session 污染**：多个 case 共享同一进程的 session 状态，可能导致 `target_sub_categories` 从一个 case 泄漏到另一个
- **品类过滤过度收紧**：Case 1/8/11 的 Turn 2-3 出现 0 cards，可能是 `target_sub_categories` 兜底逻辑过于激进
- **长对话记忆丢失**：Case 19 Turn 15-19 全部 0 cards，长对话上下文仍然脆弱

### 下一步方向

1. **修复 session 污染**：确保每个测试用例使用独立 session，不互相影响
2. **放宽品类过滤回退**：当 `target_sub_categories` 非空但无匹配时，应该回退到品类匹配（当前逻辑已修复，但 session 兜底可能引入新问题）
3. **长对话上下文压缩**：10+ 轮对话中需要将关键约束注入 LLM prompt
4. **compare_products 增强**：PC 装机场景下的对比需要返回具体组件卡片
