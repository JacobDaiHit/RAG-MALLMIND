# V3 改造前后到底提升什么：逐项对比、用户感受和验收方法

> 这份文件只比较“当前代码实际情况”和“V3 完成后应该具备的能力”。右侧是改造目标，不是当前已经实现的成绩。没有跑评测的数据，一律不能写成“已经提升了多少”。

## 1. 先说结论：V3 不是只为了让回答更像人

当前系统已经有 Router、推荐、Milvus、BM25、购物车、PC 装机等能力。V3 的重点不是再堆一个模型，而是让这些能力按正确顺序工作：

```text
旧系统容易出现的感觉：
模型、检索、目录、会话都在参与，但谁最后说了算不够清楚。

V3 的目标：
先把用户要求变成可检查条件 -> 先排不合格商品 -> 再做检索排序
-> 再用目录确认事实 -> 最后让 LLM 组织文字。
```

用户最终感受到的不是“系统用了更多术语”，而是：明确说不要的品牌不再出现；问第二个 SKU 不会答错对象；价格库存更可靠；换话题不会串；模型故障时也不会装作知道。

## 2. 总览对比表

| 方面 | 当前系统实际情况 | V3 改造后应做到什么 | 用户实际感受 |
|---|---|---|---|
| 对外工具 | 8 个工具分别暴露推荐、对比、SKU、比价、装机等 | 收敛为推荐、参数查询、购物车、闲聊 4 动作 | 功能入口一致，后端改动不容易漏分支 |
| 是否调用 LLM | `fast/balanced/full` 已不再是产品模式；但后端残留 `auto -> balanced` 兼容桥，仍间接给 LLM 开关赋值 | 先让本地生成 safety proof；无法证明简单安全才调 SemanticParse LLM | 简单卡片追问更稳更快；复杂人话仍能被理解 |
| 复杂中文 | LLM 路由和本地兜底并存，但边界分散 | 本地只处理“能完全证明”的白名单，其他默认 LLM | 不会把叙事、犹豫、代词简单当关键词 |
| 用户条件 | 条件可能散落在 prompt、路由参数、会话对象中 | 每轮都有 RequirementSpecV3，区分必须满足、尽量满足、还需追问 | 能解释为什么推荐/不推荐某商品 |
| 品牌排除 | “不要小米”可能在后续 LLM 过滤或不同路径中变弱 | 目录候选、Milvus、最终卡片三处都硬排除 | 明确排除的品牌不会又出现 |
| 商品检索 | 当前稳定预过滤字段较少；部分字段构造后未稳定写入 | 先用目录产生允许商品 ID，再在其中检索 | 不会先搜出一堆不合格商品再勉强删掉 |
| 索引数据 | chunk、schema、filter、父子层级存在不完全对齐风险 | 商品文档、证据 chunk、SKU 文档分开，字段写入可验证 | 检索行为更可预测，问题更容易定位 |
| 价格/SKU/参数 | 部分路径可能混入 RAG 或生成式回答 | 已指向商品后一律回实时目录查事实 | “第二个 512G 有吗”更不容易答错 |
| 多轮会话 | 会话对象较大，含完整结果/日志等；可能多次写 Redis | 只留卡片 ID、明确条件、focus、购物车、PC 和短期澄清草案 | 系统更不容易忘记、串话或越聊越慢 |
| 购物车 | 多分支处理目标和确认 | 统一 CartPlan + 同 session、同对象、60 秒确认 | “确认”不会误加错商品 |
| PC 装机 | 装机与普通推荐在工具层并列 | 接口合并但内部保留专用兼容性求解 | 不会因为工具简化而牺牲硬件兼容检查 |
| 出错时 | 模型或检索失败的分支较分散 | 每种依赖失败都有明确可继续范围 | 系统会诚实说“现在不能确认”，不编造 |
| 排查问题 | 很难一次看清路由、条件、候选、模型和目录版本 | TraceStore 按 request ID 记录完整链路 | 能定位到底是理解错、过滤错、检索错还是目录错 |

## 3. 最重要的五项提升，具体是怎么来的

### 3.1 提升一：明确条件终于能真正约束结果

**旧问题：** 用户说“3000 元以内、不要小米”，系统可能在不同位置各保存一次条件。若某条检索或兜底路径没有带上这两个条件，小米或超预算商品仍可能混进候选。

**V3 做法：** 把这句话写进一份 RequirementSpecV3：

```text
hard.price.max = 3000
hard.exclude_brand_family_ids = [xiaomi]
soft.desired_attributes = [camera]
```

然后让 `hard` 同时传给目录候选门、Milvus 召回前过滤和最终卡片检查。`soft` 只影响合格商品之间的排序。

**用户获得的提升：** “不要小米”不是一句礼貌性请求，而是系统不可违反的规则；“拍照优先”仍能灵活地帮助排序。

### 3.2 提升二：复杂表达不再被粗暴二选一

**旧问题：** “小米用得不好，或许华为适合我”同时有明确负面体验和带“或许”的正向猜测。若一概把它们都当成同等级过滤，就会错把华为锁死；若忽略负面，又会继续推用户明确不喜欢的品牌。

**V3 做法：** SemanticParse LLM 把它拆开：

```text
小米：已归一到 xiaomi 的直接、无条件负面体验
华为：对未来购买的弱倾向，含“或许”，不是只允许华为
商品品类：没有说清楚，需要先追问
```

LLM 的这份拆解不能自行决定过滤。随后本地 HardConstraintPromotionGate 回查原文：小米命中“已唯一归一实体 + 用着不好”的直接负面句式，写入硬黑名单；华为含“或许”，只能成为软正向偏好。只有用户明确说“小米也可以”才释放小米黑名单。

**用户获得的提升：** 系统先问“想买手机、平板还是其他产品”，不会在没有证据时擅自替用户选华为手机，也不会误把用户的犹豫当永久禁令。

### 3.3 提升三：多轮追问从“猜上下文”变成“查确定 ID”

**旧问题：** 用户问“第二个有 512G 吗”，如果系统只保留上轮大段回答或靠模型记忆，容易把“第二个”理解错，或者拿旧价格/SKU 回答。

**V3 做法：** 每张卡都有：

```text
card_id -> product_id -> default_sku_id -> catalog_version
```

SessionCore 只保留最近最多 5 张卡的简要信息。编号优先解析为卡 ID，再查实时目录。

**用户获得的提升：** 问 SKU、价格、库存、参数时，系统问的是正确商品；商品卡已过期则明确让用户重新确认，而不是猜。

同一原则也适用于澄清问题。系统会保存带 ID、答案格式和选项的短期 ClarificationPlan：用户点“平板”写入分类，点“拍照”只写软偏好，回答“3000 以内”写硬上限，只有确认整份草案时才接受“对”。因此系统不会把一个孤立的“对”交给 LLM 猜上下文，也不会把“拍照”误当成商品类目或硬过滤。

长对话额外使用 `TopicTransition`：`CONTINUE` 承接当前卡片，`SWITCH` 为“配主机”等明确新目标新开话题，`RETURN_TO_RECENT` 只在“刚才第二个手机”能唯一指向旧卡片时回跳，`NOISE_OR_CHAT` 则处理“哈哈哈/asdf qwe”等无关输入且不修改商品状态。这样闲聊和乱码不会擦掉手机上下文，新 PC 话题也不会继承手机预算、品牌黑名单或“第二个”的编号。

### 3.4 提升四：检索只在合格商品里竞争

**旧问题：** 向量检索擅长找“文字相似”，却不知道用户明确不要的品牌、库存和精确预算是否应该先淘汰。如果先搜再删，会浪费资源，也更容易漏删。

**V3 做法：** 目录候选门先构建允许商品 ID；Milvus 只在这个集合中做 dense/BM25/RRF；最终再回目录核对一次。商品文字再相似，也不能跨过硬条件。

**用户获得的提升：** 推荐更少出现明显不符合条件的商品；在“没有合适商品”时，系统会说明预算或库存问题，而不是硬塞一个不合格选项。

### 3.5 提升五：模型失败时不再假装正常

**旧问题：** 当 Router、embedding 或生成模型失败，散落的回退路径可能继续回答，但不容易判断回退后是否还遵守用户条件。

**V3 做法：** 每种失败都写明边界：

| 出问题的部分 | 可以继续 | 必须停止或澄清 |
|---|---|---|
| 语义 LLM | 精确卡片/SKU/购物车确认、明确分类请求 | 用途复杂、品牌态度复杂、品类未知的推荐 |
| embedding/Milvus | 参数/SKU/价格/比较；通过目录验证的结构化兜底 | 把旧缓存当作最新推荐 |
| 回答 LLM | 返回模板化卡片和比较表 | 编造额外卖点或事实 |
| 实时目录 | 非事实闲聊 | 确认价格、库存、SKU 是否可售 |

**用户获得的提升：** 即便系统暂时服务降级，也不会因为一句“看起来像答案”的话而被误导。

## 4. 用四个真实对话类型看差异

### 4.1 “推荐手机”

| 当前链路的风险 | V3 的处理 |
|---|---|
| 前端已没有模式选择，但后端残留 `auto -> balanced` 兼容桥，LLM 尝试仍经由这层旧开关间接决定；类目、检索条件和最终事实分别散在多个步骤 | 本地按固定顺序解析动作、分类、预算、品牌和受控属性，并要求所有业务文字都被处理；证明完整才直接构建最小需求，随后目录先过滤上架有货手机，再检索排序 |

结果：V3 不是“看到手机就不调模型”，也不是“少用了模型就更好”。只有“推荐手机”或“推荐手机、3000 内、不要小米、拍照优先”这类完整匹配受支持 grammar、枚举后的可执行**语义唯一**的句子才可直通；留下“给妈妈用”“或许”“比上一台”等未解释片段时仍调语义 LLM。`不要给我推荐手机，3000 元以内，pad不错，来点推荐` 虽可由统一词表把 `pad` 唯一归一为 `tablet`，但不匹配 V1 grammar，仍调语义 LLM 理解推荐意图；系统不会再错误追问用户想买什么品类。简单分类请求的商品范围更清楚，复杂表达也不会被关键词规则误伤。

### 4.2 “3000 内，不要小米，拍照优先”

| 当前链路的风险 | V3 的处理 |
|---|---|
| 负品牌可能后置为 LLM 软过滤；不同路径可能不一致 | 预算和品牌排除三处硬检查；拍照只在合格商品中排序 |

结果：明确条件不丢失，偏好仍有弹性。

### 4.3 “第二个 512G 有吗？多少钱”

| 当前链路的风险 | V3 的处理 |
|---|---|
| 可能依赖上轮文本、检索片段或模型上下文 | 从 session 卡片定位 product ID，直接查 SKU 和实时价格 |

结果：回答的可信度来自目录，而不是模型记忆。

### 4.4 “小米用得不好，也许华为适合我”

| 当前链路的风险 | V3 的处理 |
|---|---|
| 品牌词容易被过度当成包含/排除条件，或被完全忽略 | 先调用语义 LLM 分清实体、态度和模态词；直接无条件不喜欢写硬黑名单，“也许华为”只作软偏好；品类未知先澄清 |

结果：系统既能听懂用户态度，又不替用户做过度决定。

## 5. 哪些指标可以证明 V3 真有提升

不要只看“模型回答读起来顺不顺”。应在同一批固定测试问题、同一份目录快照上比较前后结果。

| 指标 | 怎么算 | V3 应达到的方向 |
|---|---|---|
| 硬条件违规率 | 返回卡片中违反明确预算、类目、库存、精确 ID、品牌排除的比例 | 明显下降；品牌硬排除应为 0 |
| 品牌漏放率 | 用户明确说“不要小米”后，小米家族仍出现在候选或卡片的比例 | 0 |
| 商品重复率 | 同一响应中重复 product ID 的比例 | 0 |
| SKU/价格/库存准确率 | 与同一时刻目录数据逐字段比对 | 上升；目录不可用时应转为“无法确认” |
| 多轮指代准确率 | “第几个”“这个”“前两个”等解析到正确商品的比例 | 上升；解析不了时应该澄清 |
| 误直通率 | 本应调用语义 LLM 的复杂句子却被本地规则直接处理的比例 | 接近 0 |
| 无效召回率 | 检索到后又因硬条件被删掉的商品占比 | 下降 |
| Session 大小 | Redis 每个会话的字节数和每轮写次数 | 有稳定上限，通常下降 |
| 故障安全率 | 人为让模型/检索/目录失败后，是否仍返回未验证事实 | 0 次未验证事实 |
| 时延和调用成本 | 每种动作的 P50/P95 时延、外部调用次数 | 需要实测；简单事实查询通常下降，复杂推荐不应牺牲正确性硬压时延 |

## 6. V3 不承诺什么，避免误解

1. 不承诺每一条推荐都会比旧系统更符合主观口味。V3 首先保证不违反明确条件、不编造事实。
2. 不承诺所有请求都不调 LLM。复杂用途、叙事、态度、模态、模糊指代本来就应该用 LLM 理解。
3. 不承诺只换 embedding、reranker 就能成功。目录候选门、数据写入、后检和会话契约没修好，换什么模型都可能在错误候选里排序。
4. 不承诺把含“可能/听说/如果”等限定的负面说法当硬排除；但对已唯一归一实体的直接、无条件不喜欢，V3 会写硬黑名单，直到用户明确释放该对象。
5. 不承诺工具数量少就功能少。对外动作合并，内部比较、SKU、PC 求解仍保留。

## 7. 什么时候才可以说“V3 上线成功”

只有当以下事实都能被测试或 trace 证明时，才能说 V3 成功：

- 每个本地直通请求都有 safety proof；
- 每个复杂请求都有模型解析记录和本地验证结果；
- 每个硬过滤在目录候选、Milvus、最终卡片三处一致；
- 每个商品卡、SKU、价格和参数可回到目录版本复算；
- SessionCore 不再随着聊天轮数无限膨胀；
- 任一依赖失败不会制造未经验证的商品事实；
- 同一 golden set 上，硬条件违规率、品牌漏放率、指代错误率等指标确实优于改造前。

达到这些后，再讨论排序模型、转化率和更丰富的多模态能力，才是有基础的优化。
