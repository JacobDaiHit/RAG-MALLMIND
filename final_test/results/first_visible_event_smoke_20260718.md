# 首个可显示事件冒烟结果（2026-07-18）

> 这不是 61 请求轮固定集重跑。它只验证 2026-07-18 新增的即时 `progress(stage=understanding)`
> 是否会在外部 Chat、embedding 与 Milvus 完成之前送达浏览器。

## 样本与实际链路

| 样本 | 用户输入 | 实际路由 | 实际执行器 | 首个可显示事件 | 首个业务结果 | 完整完成 |
| --- | --- | --- | --- | ---: | ---: | ---: |
| 本地直通 | 推荐手机，10000 元以内，不要小米 | `safe_direct`，`mode=product` | `v3_catalog_ranker` | 42 ms | 1154 ms | 1161 ms |
| 真实外部模型 + RAG | 送女朋友一个礼物，不知道买什么，预算 1000 元 | `semantic_executable`，`mode=explore`；DeepSeek 语义解析 6542 ms | `v3_catalog_exploration` | 19 ms | 8039 ms | 8047 ms |

两条请求的第一个事件都是：

```json
{
  "event": "progress",
  "stage": "understanding",
  "label": "正在理解需求"
}
```

它没有携带模型生成的商品、价格、SKU 或动作结论；因此即使之后 schema 校验失败、发生一次修复重试、
目录为空或请求被拒绝，也不会向用户展示未经验证的业务内容。

## 本次可替代旧“首事件”数字的口径

| 指标 | N=2 结果 | 含义 |
| --- | --- | --- |
| 首个可显示事件延迟 | mean **30.5 ms** / p50 **19 ms** / p95 **42 ms** | 用户首次看到“正在理解需求”的时间。 |
| 首个业务结果延迟 | mean **4596.5 ms** / p50 **1154 ms** / p95 **8039 ms** | 首次收到澄清、错误、文本、商品卡、事实、购物车或 PC 结果的时间。 |
| 总响应延迟 | mean **4604.0 ms** / p50 **1161 ms** / p95 **8047 ms** | 收到 `done` 的时间。 |

`N=2` 时 p50 和 p95 只是两个样本中的较小/较大值，不具有稳定的统计意义；它们只用于确认
即时事件确实先于本地路由、外部 Chat 和检索到达。后续完整固定集重跑时，`runner.py` 会自动按新口径
统计三项指标。

## 与旧 61 请求轮报告的关系

旧报告中的“首事件 mean 5558.5 ms / p50 5910 ms / p95 6500 ms”测的是**旧链路中路由完成后才发出的
第一个 SSE 事件**，不是新链路的首个可显示事件。它不能再用来描述当前用户界面的首次反馈等待时间。
旧报告仍保留其路由、约束和事实正确性结论；本次没有重跑 61 请求轮，也没有把两条冒烟样本伪装成全量延迟结论。
