# full-chain fixture 重跑前审计（2026-07-17）

## 审计原则

本 fixture 测的是当前 V3 的真实请求链路，不是要求模型猜中某个旧工具名。每条用例必须明确它是在验证：

1. 正确 action；
2. 澄清、目录范围拒绝或空候选三种失败闭环之一；
3. 推荐时真实 CandidateGate 和 embedding/Milvus；
4. 多轮时前一轮状态是否真的被下一轮正确使用。

所有回合仍要求真实 SemanticParse；推荐回合额外要求真实 embedding/Milvus evidence。SafetyProof 若未来覆盖某条 fixture，评测会明确暴露它未调用外部模型，而不是悄悄将其算通过。

## 审计结论与修改

| 范围 | 原问题 | 修正后的测试目的 |
|---|---|---|
| Case 1--8 | 用例均有明确商品/配件目标、实际目录候选及可验证的推荐结果 | 保留；验证单次语义解析、CandidateGate 和 Milvus 是否到达真实商品卡 |
| Case 9 | `配一台游戏主机` 是明确装机信号 | 保留为 PC build；验证不误走普通推荐 |
| Case 10--13 | “剪辑电脑”“RTX 游戏主机”不能直接默认装机；旧 Case 10 预期错误 | 保留模糊场景澄清；新增“配台主机/笔记本”两个后续回合，分别验证 PC 和普通推荐续接 |
| Case 14 | “预算调整与比较”中插入“换强显卡”会引入另一种合法但可能无兼容解的失败，不适合测版本比较 | 删除换显卡回合；只测 build -> 调预算 -> 比较 current/previous，并逐回合断言 action |
| Case 15 | 明确要求推荐“户外防风外套”，却期待 product type 澄清 | 改为 `catalog_scope_unsupported`；这是目录外/无对应候选的明确推荐 |
| Case 16--18 | 分别测预算过滤空候选、汽车、处方药 | 保留；三者都必须不产生虚构商品卡 |
| Case 19--26 | 缺商品类别、缺卡片引用、清空购物车和无会话卡片追问 | 保留；验证澄清与非商品副作用路径 |

## 评测器契约修正

原评测器只检查多轮案例的最终回合，因此“第一轮是否真的澄清”“第二轮是否真的继承 SessionCore”都可能被掩盖。现已支持 fixture 中的 `expected_turns`：

```text
每一条用户回合
 -> action / clarification / rejection / no_candidates 断言
 -> 该回合是否调用外部 Chat
 -> 若是推荐，是否有真实 embedding + Milvus evidence
```

Case 11、12、14 已使用该逐回合断言。全量重跑的报告会同时保留每回合 SSE 和 `turn_checks`，不再只用最终回复掩盖中间状态错误。

## 未改的边界

- 推荐商品 ID 仍采用“命中候选集合之一”的断言，而不是强制 Top-1；这样验证目录正确性，又不把合理的排序微调误判为链路失败。
- `RTX 4070` 目前不是 PC 求解器的可执行硬约束，因此该句只能测购买形式澄清。未来若支持型号约束，应新增独立 fixture，验证目录 canonical component、求解器约束和最终方案中确实含该部件。
