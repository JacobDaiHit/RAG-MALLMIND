## 三、问题分类汇总

**根因**：
- 多轮对话中，session 的 `topic_memory` 未正确传递品类上下文
- Pipeline 的 `infer_product_type` 在约束叠加时不稳定
- LLM router 在追问轮次可能忽略上文品类

### 3.2 对比能力不足（影响 5 个 case）

**表现**：用户要求对比两个产品（如"特仑苏和金典哪个好""HOKA和Nike哪个软"），Agent 要么返回空结果，要么推荐了第三个产品。

**受影响 case**：12, 15, 16, 17, 19

**根因**：
- `compare_products` 工具返回空结果
- LLM router 未正确识别对比意图

### 3.3 长对话记忆丢失（影响 3 个 case）

**表现**：在 10+ 轮对话中，Agent 丢失了用户之前明确的偏好（如品牌、型号、预算）。

**受影响 case**：14, 19, 20

**根因**：
- Session 的 `last_result` 和 `topic_memory` 在长对话中被覆盖
- LLM router 的 prompt 中上下文注入有限

### 3.4 目录缺失品类的兜底不足（影响 2 个 case）

**表现**：用户请求 catalog 中不存在的品类（如"双肩包""智能手表"），Agent 未诚实告知"没有"，而是推荐了无关商品。

**受影响 case**：10, 21

**根因**：
- Pipeline 的 `missing_subcategory` 检测未覆盖所有品类
- Agent 应在 catalog 无匹配时主动告知而非硬凑

### 3.5 PC装机多轮不稳定（影响 2 个 case）

**表现**：PC装机方案在用户追加约束（如"不要AMD""白色机箱"）后，路由从 `pc_build` 退化为 `recommend`（单品推荐）。

**受影响 case**：4, 12

**根因**：
- `is_pc_build_followup` 在某些约束追加场景下未正确识别
- 多轮PC装机的 session 上下文保持不足

## 五、总结与想法

### 5.1 整体评价

Agent 在**短上下文（2-3轮）单品类推荐**场景下表现良好，通过率约 70%。但在**多轮对话（4+轮）**和**跨品类约束叠加**场景下，品类漂移问题严重，通过率下降到约 40%。

### 5.2 核心瓶颈

**品类漂移是当前最大的能力边界**。根因链路：

```
用户追问（如"防水""续航"）
  → LLM router 将追问路由到 recommend_shopping_products
  → pipeline 的 parse_requirement 从追问文本中提取品类
  → 品类提取不稳定（"防水"可能匹配户外装备，"续航"可能匹配手机）
  → 推荐结果跳出用户原始品类
```

这不是单点 bug，而是**pipeline 在多轮场景下缺乏品类锁定机制**。

### 5.3 与之前测试的对比

- **短上下文**：与 v2_guard_downgrade 测试结果一致，表现稳定
- **多轮对话**：边界测试暴露了之前测试未覆盖的问题（品类漂移、长对话记忆丢失）
- **场景切换**：Case 21 表现超出预期，Agent 能正确处理手机→PC装机的话题切换

### 5.4 后续改进方向

1. **品类锁定机制**：在多轮对话中，当用户已明确品类（如"跑鞋"），后续追问应锁定在同一品类内，除非用户明确切换
2. **对比能力增强**：`compare_products` 工具需要能正确处理同一品类内两个具体产品的对比
3. **缺失品类兜底**：当 catalog 无匹配品类时，Agent 应诚实告知"暂无该品类"而非推荐无关商品
4. **长对话上下文压缩**：在 10+ 轮对话中，需要将关键约束（品牌、预算、品类）注入 LLM prompt，而非仅依赖 topic_memory

### 5.5 意外发现

- **场景切换能力超出预期**（Case 21）：Agent 能正确处理手机→PC装机的话题切换，说明路由层的意图识别能力较强
- **品牌排除能力稳定**（Case 6, 21）："不要苹果""不要华为"的排除过滤在之前修复后表现良好
- **购物车操作基本正确**（Case 17, 19, 20, 21）：加购操作能正确执行
