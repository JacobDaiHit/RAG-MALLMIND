### 方案 A：名称模糊匹配（fuzzy_match_cart_item）

**结论: 完整实现，超出提案设计**

实现覆盖了提案中所有设计要求：品牌匹配（"把OPPO删掉"）、标题关键词匹配（"删除Reno"）、多关键词最佳匹配（"把索尼耳机删掉"取命中率最高的商品）。源码还做了提案未提及的增强：最少 2 字符过滤防止单字误匹配、跨品类购物车匹配（美妆/食品同样适用）、空购物车和 None catalog 的防御处理。

### 方案 B：handle_cart_v2 多操作支持

**结论: 完整实现，plan+confirm 模式健壮**

`handle_cart_v2` 正确实现了 add / remove / set_quantity / clear 四种操作的分发，其中 clear 直接执行、其余走 plan+confirm 模式。`_build_confirmation_message` 根据操作类型生成不同的确认文案（移除/修改数量/加入购物车），`_make_plan` 正确存储 operation 字段和 TTL 过期时间。

测试验证（18 项全通过）：action 推断（remove/set_quantity/clear/add 关键词全覆盖）、显式 operation 优先级、无效 operation 回退到推断、端到端操作（模糊名称删除/修改数量/清空/添加）、确认文案（三种操作类型）、SSE 事件分发（clear 直接执行不走确认、remove/set_quantity 走 plan+confirm）、购物车项列表格式、plan 存储结构。

### 方案 C：追问机制（cart_clarification SSE）

**结论: 实质实现，发现一个设计边界**

歧义检测 `_check_cart_ambiguity` 正确覆盖了两种场景：序数越界（"删除第三个"但购物车只有 2 个商品时返回提示）和同品类多商品歧义（两个"智能手机"商品 + 用户说"删掉那个智能手机"时返回追问）。SSE 事件 `cart_clarification` 在歧义、空购物车、商品不存在三种场景下正确发射，且包含 `cart_items` 列表供前端渲染。

测试验证（9 项全通过）：序数越界追问、同品类歧义追问、模糊命中时无歧义、有序数时无歧义、单商品无歧义、空购物车无歧义、歧义 SSE 发射（含 cart_items）、空购物车 SSE 发射 delta、商品不存在 SSE 发射 clarification。

**发现的设计边界**（非 bug，属于设计取舍）：

1. `extract_item_index` 仅支持第 1–3 个（第一/第二/第三/1号/2号/3号），不支持"第四个"及更高序数。这意味着当购物车有 4+ 商品时，用户说"删除第五个"无法触发越界追问（系统会 fallback 到第一个商品）。这是一个已知限制而非回归。

2. 同品类歧义检测的触发条件比较严格：用户消息中不能包含任何能模糊匹配到具体商品的关键词。例如购物车有"OPPO Reno12 智能手机"和"华为 Pura 手机"，用户说"删掉智能手机"时，如果"智能手机"恰好是某商品标题的子串，模糊匹配会先命中，歧义检测被跳过。这是模糊匹配优先级高于歧义检测的架构设计决定的。

### 方案 D：set_quantity 序数支持

**结论: 完整实现，无冲突**

`apply_cart_instruction` 中 `extract_item_index` 的 guard 已移除，set_quantity 可以正确使用序数定位商品。`extract_quantity` 和 `extract_item_index` 无模式冲突——"把第二个数量改成3"能同时提取 index=1 和 quantity=3。

测试验证（8 项全通过）：序数模式识别（第一个/第二个/第三款/1号/2号）、无匹配返回 None、数量模式识别（数量改成3/改为5/修改为2/x10）、序数与数量无冲突、端到端 set_quantity（第一项/第二项/越界容错）。

### 待跟进项（非阻塞）

1. **序数覆盖范围**: `extract_item_index` 当前硬编码 1–3，建议扩展为通用数字解析（`第(\d+)个`），支持任意序数。
2. **set_quantity 静默默认**: 当用户说"改成3个"但没指定哪个商品时，系统默认操作购物车第一项。建议增加追问或至少在日志中记录 fallback 行为。
3. **歧义检测与模糊匹配的优先级**: 当前架构下，品类词（如"智能手机"）如果恰好是某商品标题的子串，会触发模糊匹配而非歧义追问。可考虑引入品类词词典来区分"名称引用"和"品类引用"。
