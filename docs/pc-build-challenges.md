方案级推荐电脑主机的最大难点在于：**它不是“找一个商品”，而是“在多个配件类目中，生成一组满足预算、用途、兼容性、库存、价格、偏好、可解释性和可购买闭环的组合方案”**。这会把普通 RAG 商品推荐问题升级成“多约束组合优化 + Agent 状态管理 + 规则校验 + 商业化展示”的问题。你们 README 里已经把 PC 整机推荐定位为主打亮点：从单品推荐升级为“可购买、可解释、可调整”的整机方案，并要求检索 CPU、GPU、主板、内存、硬盘、电源、机箱、散热等配件，同时校验兼容性、总价和替换项。

下面我按难度从底层到产品层详细拆。

---

## 1. 数据难点：PC 配件不是普通商品，字段异构且必须结构化

普通电商商品可能只需要：

```text
名称、价格、类目、品牌、图片、描述、库存
```

但 PC 配件必须有大量**结构化参数**，否则无法做兼容性判断。你们 README 中已经提到基础商品字段，以及 GPU 的扩展字段如 `chipset`、`vram`、`tdp_watt`、`length_mm`、`recommended_psu_watt`、`interface`、`compatibility_tags`、`use_case_tags` 等。

真正落地时，每个类目的字段都不同：

| 类目  | 必须字段                           | 难点                    |
| --- | ------------------------------ | --------------------- |
| CPU | socket、核心数、线程数、TDP、是否带核显、适配芯片组 | 型号命名复杂，AMD/Intel 规则不同 |
| 主板  | socket、芯片组、内存类型、M.2 数量、供电规格、板型 | 与 CPU、内存、机箱、电源都有关系    |
| GPU | 显存、功耗、长度、厚度、供电接口、性能层级          | 与机箱、电源、显示器分辨率强相关      |
| 内存  | DDR4/DDR5、容量、频率、套条数量           | 与主板、CPU 平台有关          |
| SSD | 接口、容量、PCIe 代际、散热片              | 主板 M.2 位、PCIe 通道可能限制  |
| 电源  | 额定功率、认证、接口、ATX 版本              | 不只是功率够，还要接口够          |
| 机箱  | 主板板型、显卡限长、散热限高、风扇位             | 影响主板、显卡、散热器           |
| 散热  | 风冷/水冷、高度、冷排尺寸、压制功耗             | 与 CPU TDP、机箱空间相关      |

难点不是“有没有字段”，而是**字段必须标准化**。比如同样是显卡长度，数据里可能写成：

```text
245mm
约 24.5cm
长度：245 毫米
双风扇短卡
```

如果不规范化成 `length_mm = 245`，后面无法自动判断“机箱显卡限长 320mm 是否能装”。

所以 PC 方案推荐的数据治理要比普通商品推荐更重。课题 PDF 也明确把“非结构化数据向量化、数据一致性保障、Chunking 策略设计”列为项目要求中的核心挑战。

## 2. 类目覆盖难点：少一类就不像“整机方案”

如果只是推荐 CPU + GPU，看起来像“配件建议”；如果要让评委觉得这是“整机销售方案”，至少要覆盖：

```text
CPU
主板
显卡
内存
SSD
电源
机箱
散热
```

可选扩展：

```text
显示器
键鼠
系统
无线网卡
风扇
装机服务
```

难点在于：每个类目都需要足够多的候选商品，否则方案会非常单薄。比如预算 7000 元，如果数据集中只有 2 张显卡、2 块主板、2 个电源，那么 Agent 很难做出“性价比 / 性能优先 / 静音 / 白色海景房”等不同风格方案。

你们当前 README 里把 PC 整机方案作为 Phase 3，任务包括扩充 PC 配件数据集、设计兼容性规则、实现 `pc_build_plan` 结构化输出、整机方案卡片、多轮调整和一键加购物车。这个阶段本质上会决定项目亮点能不能立住。

建议最低数据规模：

| 类目  |  最低数量 | 较理想数量 |
| --- | ----: | ----: |
| CPU |  8-12 |   20+ |
| 主板  | 10-15 |   30+ |
| GPU |  8-12 |   25+ |
| 内存  |  8-10 |   20+ |
| SSD |  8-10 |   20+ |
| 电源  |  8-10 |   20+ |
| 机箱  |  8-10 |   20+ |
| 散热  |  8-10 |   20+ |

如果时间紧，宁可每类少一点，但字段必须完整，不能只靠商品描述让大模型猜。

## 3. 兼容性规则难点：这是 PC 方案推荐的核心壁垒

普通导购推荐“蓝牙耳机”时，只要价格、品牌、续航、佩戴方式匹配就可以。但电脑主机推荐必须保证：

```text
这些配件真的能装在一起
```

兼容性校验是这个亮点最能体现工程深度的地方。你们 README 中已经列出 CPU 与主板插槽、内存类型、电源功率、显卡长度、散热规格等兼容性校验点。

### 3.1 CPU 与主板兼容

要判断：

```text
CPU socket == 主板 socket
CPU 代际是否被芯片组支持
BIOS 是否需要更新
主板供电是否足够
```

例子：

```text
Intel 14 代 CPU 不一定所有 LGA1700 主板都能直接支持，可能需要 BIOS 更新。
AMD AM5 CPU 要配 AM5 主板，不能配 AM4。
```

难点是很多规则不是简单等值匹配，而是“条件匹配”。

### 3.2 主板与内存兼容

要判断：

```text
主板支持 DDR4 还是 DDR5
最大频率支持多少
插槽数量够不够
推荐双通道还是单条
```

常见错误：

```text
DDR5 内存配到 DDR4 主板
4 条内存配到只有 2 个插槽的 ITX 主板
高频内存推荐给不稳定支持的平台
```

### 3.3 显卡与机箱兼容

要判断：

```text
显卡长度 <= 机箱显卡限长
显卡厚度是否占用过多槽位
机箱是否支持对应风道
```

这在“白色海景房”“小机箱”“ITX 主机”场景中特别重要。用户说“小一点”“放桌面上”“白色机箱”，就会触发空间和外观约束。

### 3.4 电源兼容

不能只看“推荐电源功率”。还要判断：

```text
整机估算功耗
电源额定功率
是否留 30%-40% 余量
显卡供电接口是否匹配
是否需要 12VHPWR / 12V-2x6
电源长度是否适配小机箱
```

比如一个高功耗显卡理论上 650W 能带，但如果用户后续升级、CPU 也很高功耗，推荐 750W 或 850W 会更稳。

### 3.5 散热兼容

要判断：

```text
散热器压制能力 >= CPU 发热需求
风冷高度 <= 机箱散热限高
水冷冷排尺寸 <= 机箱支持规格
```

很多 Demo 容易忽略散热，导致方案看起来像“能点亮”，但不像真实装机方案。

### 3.6 主板板型与机箱兼容

要判断：

```text
ATX / M-ATX / ITX 主板是否能装进目标机箱
```

如果用户要小机箱，就不能随便推荐 ATX 主板。

### 3.7 接口和扩展兼容

还包括：

```text
M.2 插槽数量
SATA 接口数量
Wi-Fi / 蓝牙需求
前置 Type-C 接口
显示器接口 HDMI / DP
```

如果用户说“我需要连两个 2K 显示器”，显卡和接口也要考虑。

## 4. 性能匹配难点：用户说“玩游戏”不等于知道该买什么

用户自然语言通常很模糊：

```text
我想配一台游戏电脑
预算 7000
能玩 3A
剪视频也用
要性价比高
```

Agent 要把这些话转成可计算约束：

| 用户表达  | 需要解析成                    |
| ----- | ------------------------ |
| 玩 3A  | GPU 优先，至少中高端显卡           |
| 2K 高刷 | 显卡预算占比提高                 |
| 剪视频   | CPU 多核、内存容量、SSD 容量更重要    |
| AI 入门 | 显存容量、NVIDIA CUDA 生态可能更重要 |
| 安静    | 低功耗 CPU、更好散热、更大机箱风道      |
| 小机箱   | ITX / M-ATX、显卡限长、电源 SFX  |
| 白色海景房 | 机箱、散热、显卡、内存外观统一          |

难点是不能简单关键词匹配，而要推断**用途优先级**。

比如：

```text
7000 元，2K 游戏，偶尔剪视频
```

合理预算分配可能是：

```text
GPU：35%-45%
CPU：15%-20%
主板：8%-12%
内存：6%-10%
SSD：6%-10%
电源：6%-8%
机箱散热：8%-12%
```

但如果用户说：

```text
主要剪视频，偶尔玩网游
```

预算分配就要变：

```text
CPU、内存、SSD 权重提高
GPU 权重降低
```

这部分如果做得好，评委会明显感觉“它懂装机”；如果做不好，就只是“把一堆配件拼起来”。

---

## 5. 预算控制难点：不是总价低于预算就可以

整机方案的预算控制有多个层次：

### 5.1 总价约束

必须满足：

```text
sum(所有配件价格) <= 用户预算
```

但还要考虑浮动：

```text
用户说 7000 左右，可能 6800-7300 可接受
用户说 7000 以内，就不能超
用户说越便宜越好，就要压预算
```

### 5.2 预算分配

不同用途下预算比例不同。难点在于避免极端方案：

```text
显卡太贵，电源缩水
CPU 很强，显卡太弱
机箱太贵，性能不足
电源功率刚刚够，没有余量
```

### 5.3 替换后的重新计算

多轮对话时，用户会说：

```text
显卡能不能再强一点？
再便宜 500？
换成白色机箱？
内存加到 32G？
```

每次替换都要重新计算：

```text
新总价
是否超预算
兼容性是否仍然通过
性能变化
取舍说明
购物车状态是否更新
```

这就不是一次性推荐，而是一个可迭代的方案状态机。

## 6. 检索难点：RAG 不能直接解决“组合方案”

RAG 擅长从文档或商品库里找相关内容，但 PC 整机推荐不是只召回一个商品，而是要：

```text
每个类目召回候选
每个候选做结构化过滤
候选之间做兼容性组合
组合后打分
最后交给 LLM 解释
```

所以流程应当是：

```text
用户需求解析
  -> 结构化约束提取
  -> 分类目候选召回
  -> 规则过滤
  -> 组合生成
  -> 兼容性校验
  -> 总价计算
  -> 方案打分
  -> LLM 生成解释
```

而不是：

```text
用户问一句
  -> 向量库搜索
  -> 把结果塞给 LLM
  -> LLM 自己编一套配置
```

后者一定容易幻觉。课题 PDF 明确要求 RAG 链路可靠性，严禁 AI 编造不存在的优惠券或功能，并强调价格、库存等关键参数要准确实时生效。

对于你们项目，最好把大模型定位成：

```text
解释器 + 交互器 + 需求澄清器
```

而不是：

```text
最终决策的唯一执行者
```

真正的方案生成应由规则层和检索层完成。

---

## 7. 组合爆炸难点：候选配件数量一多，排列组合会爆炸

假设每类商品只取 10 个候选：

```text
CPU 10
主板 10
GPU 10
内存 10
SSD 10
电源 10
机箱 10
散热 10
```

理论组合数量是：

```text
10^8 = 1 亿种
```

不可能暴力枚举所有方案。

所以需要剪枝策略：

### 7.1 先确定平台

比如先决定：

```text
Intel 平台 / AMD 平台
DDR4 / DDR5
ATX / M-ATX / ITX
```

平台确定后，主板和内存候选会大幅减少。

### 7.2 先锁定核心件

对于游戏主机：

```text
先选 GPU
再选 CPU
再选主板
再补齐其他配件
```

对于生产力主机：

```text
先选 CPU + 内存
再选 GPU
再补齐其他配件
```

### 7.3 候选池限流

每类只保留 Top-K：

```text
GPU Top 5
CPU Top 5
主板 Top 5
其他类目 Top 3-5
```

### 7.4 分层生成方案

不要一次生成所有组合，而是生成 2-3 种策略：

```text
性价比方案
性能优先方案
外观/静音方案
```

每种策略用不同权重打分。

## 8. 打分排序难点：什么叫“更好”的配置？

PC 方案不是简单按销量、价格或向量相似度排序。需要综合打分：

```text
score = 性能匹配分 + 预算利用分 + 兼容性分 + 品牌偏好分 + 库存分 + 可解释性分 - 风险惩罚
```

可以拆成：

| 维度   | 说明                    |
| ---- | --------------------- |
| 预算匹配 | 是否在预算内，是否合理利用预算       |
| 用途匹配 | 游戏、剪辑、AI、办公等          |
| 性能均衡 | CPU/GPU 是否有明显瓶颈       |
| 兼容性  | 是否完全通过规则校验            |
| 库存   | 是否有货                  |
| 风格   | 白色、静音、小机箱、RGB         |
| 升级空间 | 电源余量、主板扩展性            |
| 风险   | BIOS 风险、散热压制风险、机箱空间风险 |

难点是：这些指标有些来自结构化字段，有些来自经验规则，有些来自用户偏好，有些来自商品描述。必须把它们融合起来。

## 9. 防幻觉难点：LLM 很容易“装懂电脑”

这是 PC 推荐特别危险的一点。大模型可能会自信地说：

```text
这款主板支持该 CPU
这套电源足够
这张显卡适合 2K 高刷
这个机箱可以装下该显卡
```

但如果这些结论没有来自数据库或规则校验，就可能是错的。

你们 README 已经提出系统原则：只能推荐数据库中真实存在的商品，不得编造价格、库存、优惠券、参数和兼容性结论，PC 整机方案必须返回总价和兼容性检查结果。

建议落地时必须做 `validate_pc_plan()`：

```python
def validate_pc_plan(plan):
    check_all_product_ids_exist(plan)
    check_prices_match_database(plan)
    check_stock_available(plan)
    check_cpu_motherboard_socket(plan)
    check_memory_type(plan)
    check_gpu_case_length(plan)
    check_psu_wattage_margin(plan)
    check_psu_connectors(plan)
    check_cooler_case_clearance(plan)
    check_total_price(plan)
    return validation_report
```

输出时不能只写：

```text
兼容性通过
```

而要写成：

```json
"compatibility_check": {
  "status": "passed",
  "items": [
    {"rule": "CPU 与主板插槽", "result": "passed", "evidence": "CPU socket AM5，主板 socket AM5"},
    {"rule": "内存类型", "result": "passed", "evidence": "主板支持 DDR5，内存为 DDR5"},
    {"rule": "显卡长度", "result": "passed", "evidence": "显卡 245mm，机箱限长 330mm"},
    {"rule": "电源功率", "result": "passed", "evidence": "估算峰值 430W，电源 650W"}
  ]
}
```

这样答辩时非常有说服力。

## 10. 多轮对话难点：用户不是一次说清楚需求

用户会不断调整：

```text
预算 7000
主要玩黑神话悟空
不要 AMD
要白色
显卡再强一点
再便宜 500
机箱小一点
把硬盘换成 2T
这套加入购物车
```

这要求 Agent 维护一个**方案状态**：

```json
{
  "session_id": "s_001",
  "current_plan_id": "plan_001",
  "constraints": {
    "budget": 7000,
    "usage": ["2K游戏", "轻度剪辑"],
    "exclude_brands": ["AMD"],
    "color": "白色"
  },
  "selected_items": {
    "cpu": "...",
    "gpu": "...",
    "motherboard": "..."
  },
  "history": [
    {"action": "generate_plan"},
    {"action": "replace_gpu"},
    {"action": "reduce_budget"}
  ]
}
```

难点是理解“刚才那套”“第二个”“这个显卡”“再便宜点”这些指代。

课题 PDF 把多轮上下文、反选与排除、多商品对比列为 RAG 增强的加分项，并强调核心考察 Prompt、检索结果后处理，以及对否定、比较、条件组合等复杂语义的理解。

对于 PC 主机方案，复杂语义会更多：

```text
不要丐版显卡
不要太吵
不想要水冷
除了华硕还有什么
显卡不能太长
以后想升级
要能放宿舍桌面
```

这些都需要转成可执行约束。

## 11. “反选/排除”难点：否定条件必须进入检索和重排

用户说：

```text
不要 AMD
不要水冷
不要白色
不要太贵的主板
不想要海景房机箱
除了七彩虹还有什么
```

这类条件不能只在最终回答里“口头保证”，必须进入：

```text
候选过滤
重排
最终校验
```

否则会出现：

```text
用户说不要 AMD，结果推荐 AMD CPU
用户说不要水冷，结果推荐 240 水冷
用户说不要白色，结果方案里有白色机箱
```

这类错误在 Demo 中非常致命，因为评委一句追问就能暴露。

建议约束解析结果中明确分开：

```json
{
  "include": {
    "usage": ["2K游戏"],
    "color": ["black"]
  },
  "exclude": {
    "brand": ["AMD"],
    "cooling_type": ["水冷"],
    "case_style": ["海景房"]
  }
}
```

## 12. 对比解释难点：整机方案对比不是商品参数对比

普通商品对比可以比较：

```text
价格、评分、续航、材质
```

整机方案对比要比较：

```text
总价
核心性能
游戏表现
生产力表现
散热噪音
升级空间
外观风格
风险点
适合人群
```

比如输出三套方案：

```text
A：性价比游戏方案
B：性能优先方案
C：白色静音方案
```

不能只说“B 性能更强”，而要说：

```text
B 把更多预算放在显卡上，更适合 2K 高画质游戏；
A 更均衡，适合预算严格控制；
C 牺牲少量性能换外观和静音。
```

这对 LLM 的生成质量和结构化数据支持都有要求。

---

## 13. 商品卡片与整机卡片难点：UI 不能只显示一大段文字

课题要求客户端接收并渲染流式回复，回复中包含可点击商品卡片；也要求原生 App，而不是纯 Web/H5。

PC 方案更复杂，普通商品卡片不够，要有**整机方案卡片**：

```text
方案名称
总价
适合场景
兼容性状态
配件列表
每个配件的价格与理由
可替换按钮
加入购物车按钮
风险提示
```

难点包括：

1. 流式文本和结构化卡片如何同时渲染。
2. 卡片数据不能从 LLM 文本里硬解析，最好后端直接返回 `pc_build_plan` 事件。
3. 用户点击“换显卡”“降预算”“加入购物车”时，要能回到对话上下文。
4. Android 端要展示多层结构：方案卡片里嵌套配件卡片。
5. 如果商品图片加载失败，要有降级占位。
6. 价格变化或库存不足时，卡片要能刷新。

你们 README 中的 SSE 事件设计已经比较合理：`delta` 返回文本，`product_cards` 返回商品卡片，`pc_build_plan` 返回整机方案，`done` 表示结束。

---

## 14. 购物车闭环难点：整机不是一个商品，而是一组商品

如果用户说：

```text
把这套加入购物车
```

系统应该加入：

```text
CPU x1
主板 x1
显卡 x1
内存 x1
SSD x1
电源 x1
机箱 x1
散热 x1
```

而不是加入一个虚拟商品。

这会带来几个问题：

### 14.1 库存校验

加入购物车前需要检查每个配件是否有库存。

### 14.2 价格一致性

如果生成方案时总价是 6899，但加入购物车时某个配件价格变了，要提示：

```text
当前价格已变化，总价从 6899 变为 6999，是否继续？
```

### 14.3 部分缺货

如果只有散热器缺货，不能简单失败，可以提示：

```text
当前散热器缺货，我为你替换为同价位兼容款。
```

### 14.4 购物车展示

购物车里要么展示为“整机方案组”，要么展示多个配件并标记属于同一方案：

```json
{
  "cart_group": "7000元2K游戏主机方案",
  "items": [...]
}
```

课题 PDF 把购物车与下单能力列为业务闭环加分项，核心考察 Agent 能否通过对话精准操作结构化数据 CRUD，并在客户端实时反馈状态变化。

---

## 15. 实时价格与库存难点：推荐结果必须可购买

电脑配件价格波动非常大，尤其显卡、CPU、SSD。即使你们 Demo 数据是静态的，也要模拟“价格与库存一致性”。

课题 PDF 明确指出，需要确保 AI 推荐的商品价格、库存等关键参数准确且实时生效。

因此系统中至少要有：

```text
商品数据库中的当前价格
库存状态
最后更新时间
输出前校验
加入购物车前再次校验
```

不要让 LLM 直接写价格。应该由数据库提供：

```json
{
  "product_id": "gpu_002",
  "price": 2899,
  "stock": 6
}
```

LLM 只能引用这些字段。

---

## 16. 可解释性难点：评委不只看结果，还会问“为什么”

方案级推荐必须能解释：

```text
为什么选这张显卡？
为什么不是更便宜的？
为什么这个 CPU 不会拖后腿？
为什么电源选 650W？
为什么机箱能装下？
预算增加 500 先升级哪里？
预算减少 500 先砍哪里？
```

如果回答只是：

```text
这套配置性能强，性价比高
```

就不够。

更好的解释是：

```text
这套方案把预算优先分配给显卡，因为你的主要需求是 2K 3A 游戏；
CPU 选择中端多核型号，能满足轻度剪辑，同时避免挤占显卡预算；
电源选择 650W，是因为估算峰值功耗约 430W，保留约 200W 余量；
机箱显卡限长 330mm，当前显卡 245mm，可以安装。
```

这类解释要来自结构化字段和规则校验结果，而不是 LLM 自由发挥。

---

## 17. 用户需求澄清难点：信息不足时不能乱配

用户可能只说：

```text
帮我配一台电脑
```

这时如果系统直接给配置，风险很高。应该追问：

```text
主要用途是什么？
预算大概多少？
是否需要显示器？
偏向游戏、办公、剪辑还是 AI？
有没有品牌或外观偏好？
```

但追问也不能太多，否则体验差。

建议做一个“信息完整度评分”：

| 信息       | 权重 |
| -------- | -: |
| 预算       |  高 |
| 用途       |  高 |
| 是否含显示器   |  中 |
| 分辨率/目标游戏 |  中 |
| 外观/尺寸偏好  |  低 |
| 品牌排除     |  低 |

如果缺预算和用途，必须追问；如果只缺外观偏好，可以先默认生成，再提示可调整。

课题 PDF 中也强调 Agent 要能在信息不足时主动澄清，引导用户细化需求。

---

## 18. “方案级销售”难点：要像销售，不只是技术配置

真正的整机方案不是：

```text
CPU：A
显卡：B
主板：C
```

而是：

```text
这套适合谁？
解决什么需求？
为什么这样分配预算？
后续如何升级？
现在买有哪些风险？
是否能一键加购？
```

也就是说，它需要具备“销售话术 + 技术可靠性”。

优秀输出结构可以是：

```text
1. 需求复述
2. 推荐方案名称
3. 总价与预算关系
4. 配件清单卡片
5. 兼容性检查
6. 性能预期
7. 取舍说明
8. 可替换项
9. 下一步引导
```

这正好对应你们 README 中“先总结用户需求、给出推荐结论、展示商品/方案卡片、解释推荐理由、提示可调整项、给出下一步引导”的推荐回答结构。

---

## 19. 评价指标难点：如何证明推荐“好”

普通 RAG 可以看召回是否相关，但 PC 方案推荐要评估更多指标：

| 指标     | 评估方式              |
| ------ | ----------------- |
| 商品真实性  | 所有商品 ID 是否存在      |
| 价格正确性  | 输出价格是否等于数据库价格     |
| 预算正确性  | 总价是否在预算范围内        |
| 兼容性    | 规则是否全部通过          |
| 用户约束满足 | 品牌、颜色、用途、排除条件是否满足 |
| 方案合理性  | 是否性能均衡，有无明显短板     |
| 可解释性   | 推荐理由是否引用真实字段      |
| 多轮稳定性  | 替换后状态是否一致         |
| 购物车闭环  | 加购的商品是否与当前方案一致    |

建议做一个自动测试集：

```text
Case 1：7000 元 2K 游戏
Case 2：5000 元办公 + 轻度剪辑
Case 3：不要 AMD
Case 4：白色小机箱
Case 5：再便宜 500
Case 6：换显卡后加入购物车
Case 7：库存不足时降级替换
```

答辩时展示这些测试，会显得非常工程化。

---

## 20. 工程架构难点：需要规则系统，而不是只靠 Prompt

比较稳的架构应该是：

```text
LLM：理解用户、生成自然语言、追问、解释
Parser：提取预算、用途、偏好、排除条件
Retriever：按类目召回候选商品
Rule Engine：做兼容性校验
Planner：生成 1-3 套方案
Ranker：排序与打分
Validator：输出前防幻觉检查
API：返回文本流 + 结构化方案数据
Client：渲染方案卡片和购物车状态
```

也就是你们 README 里提到的：

```text
Intent Router
Constraint Parser
Retrieval Engine
Tool / Rule Layer
LLM Generator
Streaming Response to Android
```

这个设计方向是对的。

难点是每层边界要清楚。不要让 LLM 同时做所有事。

---

## 21. Demo 难点：演示路径要稳定，不能现场翻车

PC 整机推荐 Demo 最怕这些问题：

```text
LLM 输出不存在的商品
总价算错
推荐了不兼容配件
用户说换显卡后，方案状态乱了
商品卡片和文本不一致
加入购物车的不是当前方案
流式输出时卡片闪烁或顺序错乱
```

建议准备固定 Demo 脚本：

```text
用户：我要一台 7000 元以内的电脑，主要玩黑神话悟空和赛博朋克，显示器已有，2K，想要白色机箱。
Agent：生成白色 2K 游戏方案，展示总价、配件、兼容性。
用户：显卡能不能再强一点？
Agent：替换显卡，提示总价变化和性能提升。
用户：那再便宜 500。
Agent：替换机箱/SSD/主板中的某项，保证显卡不动。
用户：把这套加入购物车。
Agent：购物车展示整套配件。
```

你们 README 中也建议 Demo 展示基础导购、复杂语义、PC 整机方案、多轮调整、购物车闭环和可选试穿，这个路径符合评审看重“端到端完整性”和“创新深度”的方向。

## 22. 最容易被低估的细节难点

### 22.1 “整机主机”是否包含显示器

用户说：

```text
配一台电脑
```

可能指：

```text
只要主机
主机 + 显示器
主机 + 外设
```

必须澄清，否则预算会错。

### 22.2 用户已有配件

用户可能说：

```text
我已有 1T SSD
我已有 650W 电源
我已有 2K 显示器
```

系统要把这些作为约束，并从方案里排除或适配。

### 22.3 品牌偏好和排除

例如：

```text
不要 AMD
不要杂牌电源
主板想要华硕
显卡不要丐版
```

“丐版”这种表达很口语化，需要映射成：

```text
供电规格低
散热弱
低端系列
价格过低
用户评价较差
```

### 22.4 性能瓶颈解释

如果显卡很强但 CPU 太弱，系统要识别并提醒：

```text
这套在部分 CPU 密集游戏中可能有瓶颈。
```

### 22.5 未来升级

用户说：

```text
以后想升级显卡
```

就要选更大功率电源、更好主板、更大机箱。

### 22.6 风险提示

比如：

```text
该主板可能需要 BIOS 更新
该机箱风道一般
该散热器压制高功耗 CPU 有压力
```

这些风险提示会让方案显得专业。

## 23. 推荐你们按 P0/P1/P2 控制复杂度

### P0：必须完成，保证能演示

```text
商品数据结构化
按类目检索 CPU/GPU/主板/内存/SSD/电源/机箱/散热
总价计算
基础兼容性：CPU-主板、主板-内存、显卡-机箱、电源功率
返回 pc_build_plan
Android 展示整机方案卡片
```

### P1：体现亮点，建议重点做

```text
多轮调整：换显卡、降预算、换白色机箱
反选：不要 AMD、不要水冷、不要某品牌
方案对比：性价比 / 性能优先 / 静音外观
购物车一键加购整套
输出前 validate_pc_plan()
```

### P2：时间充足再做

```text
性能评分模型
游戏帧率预估
库存变化替换
升级路线建议
AI 试穿
拍照找配件
```

从比赛角度看，**把 P0 + P1 做扎实，比浅浅做很多功能更容易得高分**。课题 PDF 也强调评审看重深度而非广度，做精一项胜过浅尝三项。

---

## 24. 总结：PC 方案级推荐的本质难点

可以概括为 10 个关键词：

```text
异构数据
结构化参数
多类目组合
兼容性规则
预算优化
用途推理
多轮状态
反选约束
防幻觉校验
可购买闭环
```

最核心的一句话是：

**普通商品推荐是“从商品库里找一个合适的东西”；PC 整机方案推荐是“在真实商品库里，按照用户需求、预算和兼容性规则，动态组装一套可以买、能解释、可调整、可加入购物车的配置”。**

这也是它适合作为得奖级亮点的原因：它天然覆盖了课题要求里的 RAG 检索、复杂意图理解、多轮对话、反选约束、结构化数据操作、商品卡片、购物车闭环和工程可靠性。
